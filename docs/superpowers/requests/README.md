# 需求工作区

> brainstorming + writing-plans 产出的需求设计文档存放于此。
> 每个文件对应一个独立需求，实现后通过命令同步到 `specs/`。

---

## 命名规范

```
<日期>_<REQ编号>_<简短英文描述>.md
```

| 段 | 格式 | 示例 |
|----|------|------|
| 日期 | `YYYY-MM-DD` | `2026-07-20` |
| REQ编号 | `REQ-` 后接三位数字 | `REQ-017` |
| 描述 | 英文, 下划线分隔 | `add-task-filter` |

**完整示例**: `2026-07-20_REQ-017_add-task-filter.md`

---

## 文档内容模板

```markdown
# [REQ-XXX] 需求标题

**提出日期**: YYYY-MM-DD
**状态**: 设计稿 | 实现中 | 已实现 | 已废弃
**关联需求**: REQ-XXX（如有前序依赖）

## 1. 原始需求
一句话描述用户想要什么。

## 2. 验收标准
- [ ] 标准1
- [ ] 标准2

## 3. 设计方案
### 3.1 前端改动
列出需新增/修改的页面、组件。

### 3.2 后端改动
列出需新增/修改的路由、服务、中间件。

### 3.3 数据表改动
列出需新增/修改的表、字段、索引。

## 4. 实现计划
按执行顺序列出任务清单。

## 5. 变更清单
实现后由 sync-request 命令自动填充，列出实际修改的文件路径。
```

---

## 工作流

```
1. 新需求提出
       │
2. brainstorming skill  →  需求澄清 + 方案设计
       │
3. writing-plans skill  →  生成 <date>_<REQ-id>_<name>.md
       │                  存入 docs/superpowers/requests/
       │
4. 执行实现计划         →  编码开发
       │
5. ./sync-request <文件名>  →  分析 requests/ 中的 .md 文件
                              自动更新 specs/ 下对应文档
                              更新需求矩阵
                              归档或标记为已实现
```
